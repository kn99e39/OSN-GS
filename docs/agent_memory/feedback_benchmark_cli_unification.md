Do not create a new standalone report script (e.g. `phaseN_report.py`) for each new construction phase in OSN-GS's `nurbs_constructor_benchmark/`. Wire the new phase's construction logic into the SAME unified benchmark entry point (`osn-gs benchmark`, backed by `nurbs_constructor_benchmark/runner.py`) as a new `--constructor` choice instead, sharing the existing scoring/report.json machinery.

**Why:** during Phase 1-4 of [[project_boundary_first_phase1]], each phase got its own script (`component_report.py`, `phase3_report.py`, `phase4_report.py`) run directly via `python -m nurbs_constructor_benchmark.phaseN_report`, completely separate from `osn-gs benchmark` (the command the user actually runs, which only ever dispatched to `legacy`/`voxel_patch_stage1` via `TorchOSNGSPipeline`). The user ran `osn-gs benchmark` expecting it to reflect the latest (Phase 4) work, got old legacy/Stage1 output instead, and was rightly frustrated: "지금 터미널에 osn-gs benchmark 넣으면 phase 4까지 반영된... 벤치마크가 수행되는게 아니야? ... phase별로 벤치마크 나누지 말고 기존 벤치마크를 대체하는 식으로 작업해야지." This was corrected by adding a `boundary_first` constructor option to `runner.py` itself and deleting the three standalone scripts.

**How to apply:** whenever a new construction/fitting phase is implemented anywhere in this project (OSN-GS), before considering it "done," check whether it's reachable from the actual CLI entry point the user runs (`osn-gs <command>`, `osn_gs/cli.py`). A benchmark, report, or diagnostic script that only a person who already knows its exact module path can invoke is effectively invisible to the user and will produce this exact complaint again. This applies beyond just `nurbs_constructor_benchmark/` — the same principle holds for `osn-gs train`, `osn-gs inspect-surface`, etc. This is distinct from the "benchmark-only, don't touch the trainer" constraint in [[project_boundary_first_phase1]]: unifying the benchmark CLI surface does not require touching `osn_gs/core/torch_pipeline.py`'s trainer-facing default or `torch_trainer.py` — those stay separate concerns.

**Round 2, 2026-07-20 — "unified" still silently ran legacy by default.** After Phase 1-4 landed and the benchmark was "unified" (worklog 32/37), `runner.py`'s `--constructor` flag was added but its `default=` was left as `"legacy"`. So plain `osn-gs benchmark` (no flags — exactly what the user types) still ran legacy under the hood while every worklog/memory claimed unification was complete. The user caught this by asking me to verify directly rather than trust the docs ("문서만 맹신하지 말고 네가 직접 검증해봐") — I ran the plain command, diffed `report.json`'s `nurbs_constructor_mode` field and the actual numbers against the known boundary_first baseline, and confirmed it was still legacy. Fixed by removing `"legacy"` from `--constructor`'s choices entirely and flipping the default to `"boundary_first"` (`voxel_patch_stage1` stays selectable as the Stage 1 ablation baseline). This also required fixing `scripts/stage1_ablation.py`, which had a `"legacy": ([], True)` row relying on "empty args = legacy" — with the new default that row would have silently benchmarked boundary_first while still labeling itself "legacy" in the output table.

**Standing lesson: "wired into the CLI" is not the same as "is what actually runs by default."** When a new phase is unified into an existing CLI entry point, also check the flag's *default value*, not just that the choice exists. And more generally: when a memory or worklog claims some integration/consolidation is "done," verify it by actually running the command the user runs and inspecting real output — don't repeat the claim back to the user as fact. See [[project_osn_gs_direction]] for the standing "no legacy fallback preference" context and [[project_boundary_first_phase1]] §10.1 for the plan's own eventual full-legacy-removal end-state (trainer default, `torch_pipeline.py`) — user confirmed 2026-07-20 this full removal should actually happen once the boundary-first work fully matures, not stay a background policy line.
